# CO-001: Per-Endpoint Rate Limits

**Date:** 2026-03-10
**SOW Ref:** SOW-001 §5 Exclusion G1
**Status:** APPROVED — work authorized

---

## 1. Request

The current gateway applies a single global rate limit (default 60 req/min per IP) across all endpoints. This means:

- A burst of static page loads can exhaust a user's rate budget before they make a single API call.
- Expensive operations (chat completions, token generation) share the same budget as cheap health checks.
- An attacker can consume the global budget with cheap requests, starving legitimate expensive ones.

Per-endpoint rate limits allow different ceilings for different cost profiles, improving both UX and security posture.

**Trigger:** Operator feedback that global limits are too blunt for production workloads with mixed traffic patterns.

## 4. Design

```
ENDPOINT_RATE_LIMITS="/api/v1/chat=10;/api/v1/models=30;/admin/=5"
```

- Parsed at startup into `[]EndpointLimit{Prefix, Limit}` sorted longest-prefix-first
- Matching: longest prefix wins (e.g., `/api/v1/chat` before `/api/v1/`)
- Each prefix gets its own `map[string]*ipRecord` with the global window
- If no prefix matches → falls through to global `RateLimiter.Allow(ip)`
- Middleware inserted **before** global rate limiter in the chain:
  `EndpointRateLimit → RateLimitWithGuard (global fallback)`

```
Request
  → EndpointRateLimit (checks per-endpoint map)
    → if matched prefix: use endpoint-specific limit
    → if no match: pass through (global limiter handles it)
  → RateLimitWithGuard (global fallback + brute-force)
  → Auth → BodyScanner → Proxy
```
