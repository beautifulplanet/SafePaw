## 1. Purpose

This Statement of Work governs the delivery of SafePaw, a security gateway and administration layer for the OpenClaw AI assistant. It documents what has been built, what remains, the acceptance criteria for each deliverable, and the consequences of action and inaction for every scope item.

This is not a sprint plan. It is a contract-grade scope document modeled after construction industry practice: a Schedule of Values (what's delivered), a Punch List (what's not yet right), and a formal Change Order process for anything new.

## 4. Phase 2 Scope — Next Deliverables

Each item below includes acceptance criteria and a "Then What?" consequence analysis.

### 4.1 PL1 — Non-English Injection Patterns ✅ CLOSED

**Deliverable:** Add prompt injection detection for the top 5 non-English languages by internet usage (Mandarin, Spanish, Arabic, Hindi, Japanese).

**Acceptance criteria:**
- [x] At least 5 injection patterns per language (25+ new patterns total)
- [x] Regression test suite with golden attack strings per language
- [x] False-positive rate validated against benign multilingual text corpus
- [x] gosec + tests pass, coverage does not regress

**v4.0.0 upgrade** (commit 4831d01): Expanded to 12 languages, added language-agnostic defense:
- [x] Korean, Russian, Portuguese, French, German, Turkish patterns added
- [x] ChatML format injection detection (`<|im_start|>system`)
- [x] JSON messages-array injection detection (`"role":"system","content":...`)
- [x] Unicode script analysis: detects unsupported scripts + English injection keywords
- [x] 186 test cases total (was 45 → 103 → 186), coverage 83.3%

**Defense-in-depth architecture** (documented in sanitize.go):

| Layer | What it catches | What it misses |
|-------|----------------|----------------|
| 1. Regex patterns (12 languages) | Known attack phrases in covered languages | Novel phrasings, uncovered languages |
| 2. Structural detection | ChatML, JSON role injection, delimiters | Natural language attacks without structural markers |
| 3. Unicode script analysis | Unsupported scripts (Thai, Bengali, etc.) mixed with English injection keywords | Pure unsupported-script attacks with zero English |
| 4. Context stuffing detection (v5.0.0) | Messages >32KB designed to push system prompt out of context window | Short messages that pack meaning densely |
| 5. Adversarial suffix detection (v5.0.0) | GCG attack strings (high entropy, long symbol runs) | Low-entropy adversarial tokens |
| 6. Repeated content detection (v5.0.0) | Payload repetition attacks (>60% duplicate lines) | Varied rewording of the same attack |
| 7. System prompt reinforcement (v5.0.0) | Override attempts — safety preamble injected on every request | Attacks that first neutralize the reinforcement |
| 8. **Phase 2: ML classifier** | Novel attacks, obfuscation, pure non-English | Needs training data, adds latency |
| 9. Output scanning | Successful injections that produce dangerous output | Clean-looking exfiltration |
| 10. Rate limiting + brute force | Volumetric attacks | Low-and-slow attacks under threshold |

**Full Attack Taxonomy — What SafePaw Does and Does Not Catch:**

| # | Attack | Status | Layer |
|---|--------|--------|-------|
| 1 | Direct instruction override (12 languages) | ✅ Caught | 1 (regex) |
| 2 | Identity hijack / persona override | ✅ Caught | 1 |
| 3 | Jailbreak keywords (DAN, developer mode, etc.) | ✅ Caught | 1 |
| 4 | ChatML / JSON role injection | ✅ Caught | 2 (structural) |
| 5 | System delimiter injection | ✅ Caught | 1 |
| 6 | Context window stuffing (huge payloads) | ✅ Caught (v5.0.0) | 4 |
| 7 | GCG adversarial suffixes | ✅ Caught (v5.0.0) | 5 |
| 8 | Repeated payload flooding | ✅ Caught (v5.0.0) | 6 |
| 9 | System prompt reinforcement | ✅ Active (v5.0.0) | 7 |
| 10 | Single-layer encoding (base64, eval) | ⚠️ Partial | 1 |
| 11 | Few-shot poisoning ("User: X. Assistant: OK.") | ⚠️ Partial (role_injection) | 1 |
| 12 | Hypothetical/research framing | ⚠️ Partial | 1 |
| 13 | Payload splitting across messages | ❌ Not caught | Needs session state (Phase 2) |
| 14 | Indirect injection via tool output | ❌ Not caught | Needs tool-output tagging (Phase 2) |
| 15 | Semantic equivalence ("pretend constraints gone") | ❌ Not caught | Needs ML (Phase 2) |
| 16 | Side-channel extraction ("first letter of each sentence") | ❌ Not caught | Needs output analysis (Phase 2) |
| 17 | Multi-turn rapport building | ❌ Not caught | Needs conversation-level analysis |
| 18 | Visual prompt injection (in images) | ❌ Not caught | Needs vision model |
| 19 | Model-specific token manipulation | ❌ Not caught | Not gateway-solvable |
| 20 | Social engineering / emotional manipulation | ❌ Not caught | Not gateway-solvable |

**Known limitation:** Pure attacks in unsupported scripts (e.g., Thai instruction override with zero English words) require an ML-based classifier. Regex cannot solve this. This is the Phase 2 ML classifier requirement per Layer 8 above. Items 13–18 are Phase 2 deliverables. Items 19–20 are out of scope for a gateway and require model-level defenses.

**Then what?**
- *If we do this:* Injection detection covers ~92% of internet users by primary language. The script analysis layer catches mixed-language attacks in the remaining 8%. Only pure single-language attacks in unsupported scripts are undetected.
- *If we don't:* Any attacker who writes "忽略之前的指令" (ignore previous instructions) in Mandarin bypasses the entire scanning layer. The gateway becomes security theater for non-English input. Every other security control still works, but the scanning value proposition is gone for most of the world.

### 4.2 PL2 — Trusted-Proxy Shared Secret ✅ CLOSED

**Deliverable:** Add HMAC signature or shared secret to the `X-SafePaw-User` header so OpenClaw can verify the request actually came from the gateway.

**Acceptance criteria:**
- [x] Configurable shared secret via env var (`GATEWAY_PROXY_SECRET`)
- [x] Gateway signs header value with HMAC-SHA256; OpenClaw verifies
- [x] OpenClaw rejects requests with missing or invalid signature
- [x] Documented in SECURITY.md and deployment guide

**Then what?**
- *If we do this:* A compromised container on safepaw-internal cannot impersonate the gateway. Defense-in-depth improves. Deployment complexity increases slightly (one more secret to manage and rotate).
- *If we don't:* Docker network isolation is the sole protection. If an attacker escapes a container or a new service is added to the network carelessly, they get unauthenticated access to OpenClaw as `safepaw-gateway`. The risk is contained as long as the network stays clean — but "as long as" is not a security guarantee.

### 4.3 PL3 — Autonomous Browsing Threat Documentation ✅ CLOSED

**Deliverable:** Add threat G10 to THREAT-MODEL.md covering AI autonomous browsing / visual prompt injection.

**Acceptance criteria:**
- [x] New threat entry in STRIDE section (Information Disclosure + Tampering)
- [x] Documents attack vector: attacker hosts page → AI browses → injected instructions
- [x] Documents QR code / encoded file variant (visual prompt injection)
- [x] Recommends mitigations: URL allowlisting, sandboxed browsing, content scanning of fetched pages
- [x] References existing sanitizer as partial mitigation

**Then what?**
- *If we do this:* Operators and future developers are aware of the threat. The threat model becomes a living document that guides feature decisions. If OpenClaw adds a browsing tool, the team knows to gate it.
- *If we don't:* The threat model has a blind spot. Someone enables browsing. An attacker hosts a page with instructions. The AI follows them. Nobody saw it coming because nobody wrote it down.

### 4.4 PL4 — Persistent Tamper-Evident Audit Ledger ✅ CLOSED

**Deliverable:** Replace the in-memory receipt ring buffer with a persistent, hash-chained NDJSON file.

**Acceptance criteria:**
- [x] Each entry includes SHA-256 hash of previous entry (hash chain)
- [x] Written to a mounted volume (survives container restart)
- [x] Verification function `VerifyLedgerFile()` detects tampering
- [x] Automatic rotation when file exceeds 100MB
- [x] Existing receipt API continues to work (in-memory ring buffer + file backend)
- [x] Tests cover: append, chain verification, tamper detection, rotation

**Then what?**
- *If we do this:* Incident forensics have a durable, tamper-evident trail. SOC2 auditors can verify integrity. Disk usage increases (~50-100 bytes per request). Rotation and retention policy needed. If the volume fills up and nobody monitors it, writes fail — need an alert.
- *If we don't:* A container restart erases all evidence. An attacker who can restart the gateway covers their tracks completely. Compliance audits will flag the gap. The existing ring buffer is useful for real-time debugging but useless for forensics.

**Acceptance criteria:**

Every support request will be "I installed it but nothing works" — because the wizard doesn't verify that it works.

## 6. Change Order Process

Any item not listed in §4 requires a **Change Order** before work begins.

### Process:

1. **Request** — Describe the item, why it's needed, and what triggered it
2. **Impact assessment** — What does it touch? What does it delay? What are the consequences?
3. **Acceptance criteria** — Specific, testable conditions for "done"
4. **Approval** — Owner signs off before any code is written
5. **Record** — Filed as `docs/scope/CO-NNN.md` with sequential numbering

### Rules:

- No work starts on out-of-scope items without a Change Order
- "While I'm in there" additions require a Change Order if they touch more than the file being edited
- Bug fixes to delivered baseline (§2) do **not** require a Change Order
- Security vulnerabilities can be fixed immediately and documented as a Change Order after the fact
