## 1. Purpose

This Statement of Work governs the delivery of SafePaw, a security gateway and administration layer for the OpenClaw AI assistant. It documents what has been built, what remains, the acceptance criteria for each deliverable, and the consequences of action and inaction for every scope item.

This is not a sprint plan. It is a contract-grade scope document modeled after construction industry practice: a Schedule of Values (what's delivered), a Punch List (what's not yet right), and a formal Change Order process for anything new.

## 3. Punch List — Known Deficiencies in Delivered Work

Items below are defects or gaps in the baseline that must be closed before the baseline is considered "substantially complete." These are not new features — they are things the baseline should have had.

| PL# | Deficiency | Severity | Consequence if not fixed |
|-----|-----------|----------|--------------------------|
| PL1 | **Non-English injection patterns missing** — ~~Sanitizer only detects English-language prompt injection.~~ **CLOSED** (commit 9d832ef): 25 patterns added for Mandarin, Spanish, Arabic, Hindi, Japanese. Regression suite: 25 attack payloads + 13 benign strings, zero false positives. | ~~High~~ Done | Any non-English-speaking attacker bypasses injection detection entirely. ~60% of internet users are non-English. This is not an edge case — it's a primary attack surface left uncovered. |
| PL2 | **X-SafePaw-User header has no shared secret** — Gateway sends a plain header to identify itself to OpenClaw. Any container on the Docker network can spoof it. | Medium | An attacker who compromises any container on safepaw-internal can impersonate the gateway to OpenClaw, bypassing all auth/rate/scan controls. Docker network isolation is the only protection. If that fails, everything fails. |
| PL3 | **Autonomous browsing not documented as threat** — OpenClaw can browse the web autonomously. THREAT-MODEL.md does not address the risk of AI navigating to attacker-controlled pages containing prompt injection. | Medium | The threat model is incomplete. If OpenClaw gains browsing capability (or already has it), an attacker can host a page that injects instructions into the AI's context. Without documentation, operators won't know to restrict this. |
| PL4 | **Audit log is ephemeral** — Receipt ledger is an in-memory ring buffer. No persistent, tamper-evident storage. | Medium | An attacker who gains container access can restart the gateway to erase all evidence. Forensic analysis after an incident has no durable trail. | PL5 | **ws_proxy.go coverage at 39%** — WebSocket proxy is the most security-critical path and the least tested. | Medium | WebSocket is where all AI conversation traffic flows. The proxy has already had 3 bugs this session (hijack, origin, errcheck). Low coverage means more latent bugs. |

## 4. Phase 2 Scope — Next Deliverables

Each item below includes acceptance criteria and a "Then What?" consequence analysis.

### 4.1 PL1 — Non-English Injection Patterns

**Deliverable:** Add prompt injection detection for the top 5 non-English languages by internet usage (Mandarin, Spanish, Arabic, Hindi, Japanese).

**Acceptance criteria:**
- [x] At least 5 injection patterns per language (25+ new patterns total)
- [x] Regression test suite with golden attack strings per language
- [x] False-positive rate validated against benign multilingual text corpus
- [x] gosec + tests pass, coverage does not regress

**Then what?**
- *If we do this:* Injection detection covers ~85% of internet users by primary language. False positives will appear — especially with short CJK phrases that resemble commands. The regression suite catches regressions. Maintenance cost: patterns need periodic review as attack techniques evolve.
- *If we don't:* Any attacker who writes "忽略之前的指令" (ignore previous instructions) in Mandarin bypasses the entire scanning layer. The gateway becomes security theater for non-English input. Every other security control still works, but the scanning value proposition is gone for most of the world.

---

### 4.2 PL2 — Trusted-Proxy Shared Secret

**Deliverable:** Add HMAC signature or shared secret to the `X-SafePaw-User` header so OpenClaw can verify the request actually came from the gateway.

**Acceptance criteria:**
- [ ] Configurable shared secret via env var (`GATEWAY_PROXY_SECRET`)
- [ ] Gateway signs header value with HMAC-SHA256; OpenClaw verifies
- [ ] OpenClaw rejects requests with missing or invalid signature
- [ ] Documented in SECURITY.md and deployment guide

**Then what?**
- *If we do this:* A compromised container on safepaw-internal cannot impersonate the gateway. Defense-in-depth improves. Deployment complexity increases slightly (one more secret to manage and rotate).
- *If we don't:* Docker network isolation is the sole protection. If an attacker escapes a container or a new service is added to the network carelessly, they get unauthenticated access to OpenClaw as `safepaw-gateway`. The risk is contained as long as the network stays clean — but "as long as" is not a security guarantee.

---

### 4.3 PL3 — Autonomous Browsing Threat Documentation

**Deliverable:** Add threat G10 to THREAT-MODEL.md covering AI autonomous browsing / visual prompt injection.

**Acceptance criteria:**
- [ ] New threat entry in STRIDE section (Information Disclosure + Tampering)
- [ ] Documents attack vector: attacker hosts page → AI browses → injected instructions
- [ ] Documents QR code / encoded file variant (visual prompt injection)
- [ ] Recommends mitigations: URL allowlisting, sandboxed browsing, content scanning of fetched pages
- [ ] References existing sanitizer as partial mitigation

**Then what?**
- *If we do this:* Operators and future developers are aware of the threat. The threat model becomes a living document that guides feature decisions. If OpenClaw adds a browsing tool, the team knows to gate it.
- *If we don't:* The threat model has a blind spot. Someone enables browsing. An attacker hosts a page with instructions. The AI follows them. Nobody saw it coming because nobody wrote it down.

---

### 4.4 PL4 — Persistent Tamper-Evident Audit Ledger

**Deliverable:** Replace the in-memory receipt ring buffer with a persistent, hash-chained NDJSON file.

**Acceptance criteria:**
- [ ] Each entry includes SHA-256 hash of previous entry (hash chain)
- [ ] Written to a mounted volume (survives container restart)
- [ ] Verification command (`safepaw verify-ledger`) detects tampering
- [ ] Automatic rotation when file exceeds 100MB
- [ ] Existing receipt API continues to work (reads from file instead of memory)
- [ ] Tests cover: append, chain verification, tamper detection, rotation

**Then what?**
- *If we do this:* Incident forensics have a durable, tamper-evident trail. SOC2 auditors can verify integrity. Disk usage increases (~50-100 bytes per request). Rotation and retention policy needed. If the volume fills up and nobody monitors it, writes fail — need an alert.
- *If we don't:* A container restart erases all evidence. An attacker who can restart the gateway covers their tracks completely. Compliance audits will flag the gap. The existing ring buffer is useful for real-time debugging but useless for forensics.

---

### 4.5 PL5 — WebSocket Proxy Test Coverage

**Deliverable:** Increase ws_proxy.go test coverage from 39% to 75%+.

**Acceptance criteria:**
- [ ] Tests cover: successful upgrade, auth rejection, origin validation, header injection, backend unavailable, connection timeout, large message, concurrent connections
- [ ] Coverage ≥ 75% for ws_proxy.go specifically
- [ ] No new dependencies (use httptest for WS testing)
- [ ] All existing tests continue to pass

**Then what?**
- *If we do this:* The most security-critical code path has proportional test coverage. Future changes to the WS proxy are caught by tests instead of by users reporting black screens. Cost: writing WS tests is tricky (need real upgrades, concurrent reads/writes).
- *If we don't:* The WS proxy continues to be the most fragile component. Three bugs were found manually this session. There are likely more. Every future change to the proxy is a gamble without test coverage.

---

### 4.6 Wizard Onboarding Overhaul

**Deliverable:** Replace the current 4-step wizard (Welcome → API Key → Security → Done) with a guided onboarding flow.

**Acceptance criteria:**
- [ ] New flow: Who are you? → What will you do? → Pick your AI → Security level → Verify it works
- [ ] "Verify it works" step makes a real round-trip through gateway → OpenClaw and shows success/failure
- [ ] Each step has a plain-English explanation of what it means and why it matters
- [ ] Existing config values are preserved (upgrade-safe)
- [ ] Wizard tests updated for new flow

**Then what?**
- *If we do this:* A non-technical user can set up SafePaw without reading documentation. The verification step catches misconfigurations at setup time instead of as a "Chat with AI is a black screen" bug report. Onboarding becomes self-documenting. Cost: significant UI/UX work, needs design decisions (what AI providers to list, what security levels mean).
- *If we don't:* The current wizard gets users from zero to "Docker containers are running" but not to "I can talk to the AI." The gap between "installed" and "working" is where users give up. Every support request will be "I installed it but nothing works" — because the wizard doesn't verify that it works.

## 6. Change Order Process

Any item not listed in §4 requires a **Change Order** before work begins.

### Process:

1. **Request** — Describe the item, why it's needed, and what triggered it
2. **Impact assessment** — What does it touch? What does it delay? What are the consequences?
3. **Acceptance criteria** — Specific, testable conditions for "done"
4. **Approval** — Owner signs off before any code is written
5. **Record** — Filed as `docs/scope/CO-NNN.md` with sequential numbering

### Rules:

- No work starts on out-of-scope items without a Change Order
- "While I'm in there" additions require a Change Order if they touch more than the file being edited
- Bug fixes to delivered baseline (§2) do **not** require a Change Order
- Security vulnerabilities can be fixed immediately and documented as a Change Order after the fact

## 7. Acceptance & Sign-Off

### Substantial Completion

The baseline (§2) is considered substantially complete when:
- All Punch List items (§3) are either closed or formally deferred via Change Order
- CI pipeline is green on `main`
- All acceptance criteria in §4 are met for items that have been started

### Final Acceptance

Phase 2 (§4) is complete when:
- Every item in §4 has all acceptance criteria checked off
- No new Punch List items have been added without a plan to address them
- Owner has reviewed and accepted each deliverable
