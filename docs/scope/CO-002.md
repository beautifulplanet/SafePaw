# CO-002: Playwright E2E Tests — Wizard UI

**Date:** 2026-03-10
**SOW Ref:** SOW-001 §5 Exclusion T3
**Status:** DONE — all acceptance criteria verified

---

## 1. Request

The Wizard UI (React 19 + Vite) has Go-side unit tests for API handlers but zero
browser-level E2E tests. Critical user flows — login, navigation, configuration —
are untested from the browser perspective. JavaScript rendering bugs, CSRF cookie
handling, and session flow regressions would go undetected.

**Trigger:** Coverage milestone reached (80%+). Time to widen the safety net from
API-level to browser-level.

## 2. Impact Assessment

### Files touched:
| File | Change |
|------|--------|
| `services/wizard/ui/package.json` | Add Playwright devDependency + test script |
| `services/wizard/ui/playwright.config.ts` | Playwright configuration (base URL, retries) |
| `services/wizard/ui/e2e/*.spec.ts` | E2E test files |

### What it does NOT touch:
- Go code (no changes to handlers, middleware, or config)
- CI pipeline (Playwright CI integration is a separate future CO)
- Production bundle (Playwright is devDependency only)

### Risk: Low
- Additive — no production code changes
- Playwright runs against the existing Go server
- Tests run locally; CI integration deferred to avoid Docker-in-Docker complexity

## 3. Acceptance Criteria

- [x] Playwright installed as devDependency in `services/wizard/ui/`
- [x] `playwright.config.ts` configured for local wizard server (:3099, webServer auto-start)
- [x] E2E test: login page renders (title, password field, submit button visible)
- [x] E2E test: login with wrong password shows error
- [x] E2E test: login with correct password navigates away from login
- [x] E2E test: logout returns to login page
- [x] `npm run test:e2e` script runs tests
- [x] Tests pass against running wizard server (4/4 green, 15s)

## 4. Design

### Test architecture:
```
wizard Go binary (port 3000)
  ↑
Playwright (Chromium headless)
  ↑
e2e/*.spec.ts
```

### Scope (deliberately minimal):
- **login.spec.ts** — Login flow: render, invalid password, valid login, logout
- Server started manually or by test harness (`WIZARD_ADMIN_PASSWORD=test-password`)
- No Docker-dependent pages tested (prerequisites, status) — those need Docker mocks

### Why not test everything:
Per AGENTS.md: "Walk, don't sprint." Login is the critical gate. If login works
end-to-end (React → fetch → Go handler → session cookie → redirect), the plumbing
is validated. Page-specific tests come in future COs when Docker mocking is available.
